## 二、检测流水线与数据格式

实验中的每一帧原始图像形状为：

```text
480 × 640 × 3
```

其中 `3` 代表 RGB 三个颜色通道。进入检测器前，需要转换为模型常用的输入格式：

```text
resize → normalize → HWC 转 NCHW → inference
```

含义如下：

| 步骤 | 含义 |
|------|------|
| `resize` | 将不同尺寸的原始图像缩放到统一的 `320 × 320` |
| `normalize` | 像素从 `0~255` 转换到 `0.0~1.0` 的浮点数 |
| HWC | 图像原始排列：高、宽、通道，例如 `(320, 320, 3)` |
| NCHW | 深度学习模型常用排列：批次、通道、高、宽，例如 `(1, 3, 320, 320)` |
| inference | 将整理好的张量送入检测网络得到结果 |

## 三、基线方案与优化方案

### 1. 基线方案

基线方案顺序处理 80 帧。每处理一帧，都会创建多个新的数组：缩放结果、浮点归一化结果、NCHW 输入张量、特征图和激活结果。

```python
resized = frame[...]
normalized = resized.astype(np.float32) / 255.0
tensor = np.transpose(normalized, (2, 0, 1))[np.newaxis, ...].copy()
```

这种做法逻辑直接，但频繁分配和回收大数组，会增加内存管理开销，也可能造成最大延迟尖峰。

### 2. 优化方案

优化方案结合了课程 PDF 的四个方向。

| 优化项 | 代码中的做法 | 目的 |
|--------|--------------|------|
| 预分配缓冲区 | 每个线程只在第一次使用时创建 `input_buf` 和 `feature_buf` | 避免每帧重复申请内存 |
| 原地运算 | `np.multiply(..., out=input_buf)` 等 | 将结果直接写入已有数组，减少中间对象 |
| 预热 | 正式测试前先处理 5 帧 | 排除首次分配、线程启动等冷启动影响 |
| 多工作线程 | `ThreadPoolExecutor(max_workers=4)` | 多帧可独立处理时，提高整体吞吐量 |

在真实 ONNX Runtime 项目中，对应做法是：

```python
opts.intra_op_num_threads = 4
opts.inter_op_num_threads = 2
opts.graph_optimization_level = ort.GraphOptimizationLevel.ORT_ENABLE_ALL
```

本次没有实际 `.onnx` 文件，所以没有调用 ONNX Runtime；但预分配、预热、线程并行和减少中间数组的优化思想与课程代码一致。

## 四、为什么要预热

程序第一次运行某个模块时，可能额外发生下列事情：

1. 创建线程；
2. 分配内存；
3. 初始化运行时；
4. 创建缓存；
5. 对首次调用进行内部准备。

若直接把第一次调用计入正式成绩，结果会偏慢且不稳定。因此代码先运行 5 次预热：

```python
list(pool.map(detector.detect, frames[:WARMUP_RUNS]))
```

预热不让算法本身变快，而是让后续基准测试更接近设备稳定运行时的表现。

## 五、如何理解两个不同的性能指标

并行系统中必须区分“单帧任务时间”和“整体吞吐量”。

| 指标 | 问题 | 本次用途 |
|------|------|----------|
| 单任务延迟 | 一帧在某个工作线程中实际执行多久？ | 对比单帧的 Mean、P95、Max |
| 吞吐量 | 一批帧全部完成得有多快？ | 评价多线程是否提高系统处理能力 |

若 4 个线程同时工作，一帧在其中一个线程内仍可能要约 1.5 ms，但 4 帧可以同时向前推进。因此整体“平均每帧墙钟时间”可接近 0.4 ms。这不是说单帧真的只计算了 0.4 ms，而是说从系统整体看，完成一帧平均只占用这么多总时间。

```text
单任务延迟：关注一帧是否及时完成
总体吞吐量：关注一秒能完成多少帧
```

```text

```

这一步很重要。性能优化不能只追求更快；如果优化后检测结果改变，速度再高也不能作为有效优化成果。

## 八、与真实 YOLOv10 + ONNX Runtime 的对应关系

| 本次教学版本 | 部署真实 YOLOv10 时的对应部分 |
|--------------|-------------------------------|
| 合成 `inference` | `session.run(None, {input_name: input_tensor})` |
| 预分配 `input_buf` | 固定形状的 NCHW 输入工作缓冲区 |
| 原地归一化 | 预处理中避免反复创建 `float32` 临时数组 |
| 工作线程池 | 摄像头采集、预处理、推理、后处理的生产者/消费者流水线 |
| 预热 5 次 | ONNX Runtime 初始化、内存池和模型执行路径预热 |
| 结果一致性检查 | 比较优化前后检测框、类别和置信度是否满足误差要求 |

真实部署还需要加入：模型文件管理、摄像头或视频输入、后处理 NMS、模型准确率验证、内存占用测量以及针对目标硬件的线程数调优。

## 九、今日总结

Day 15 完成了第三周的综合性能优化练习。通过预分配工作缓冲区、原地运算、预热和 4 线程并行处理，在保持基线与优化结果一致的前提下，将整体检测流水线吞吐提高到 **4.06 倍**，超过课程要求的 2 倍目标。

实验也说明了一个重要原则：优化前先建立基线，优化后同时检查平均延迟、P95、最大延迟、总体吞吐量和结果正确性。只看一个“更快”的数字不够；实时嵌入式系统还需要关注偶发尖峰，以及优化是否改变系统功能。
