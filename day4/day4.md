## 嵌入式硬件方向实习 · Day 4 学习笔记

### 一、今日主题
**多线程视频处理与检测系统**：生产者-消费者模式异步视频捕获 + YOLOv10-N 检测器集成（Letterbox 预处理 + 坐标映射）

---

### 二、上午：多线程实时视频捕获

**1. 生产者-消费者模式**
- 将视频采集与推理处理解耦为两个独立线程：
  - **生产者线程**：持续从摄像头读取帧，写入有限长度队列。
  - **主线程（消费者）**：从队列取帧，执行预处理与推理，显示结果。
- 优势：采集不等待推理，推理不阻塞采集，两者并行运行。

**2. 帧丢弃策略（保持实时性）**
- 队列深度设为 **3 帧**（`buf_size=3`）。
- 当队列满时，丢弃最旧的帧，放入最新帧。
- 效果：若推理速度慢于采集速度，系统会"跳帧"，但始终处理最新的画面，避免延迟累积。

**3. 端到端延迟分析**
- 在 `Frame` 数据结构中记录采集时间戳 `timestamp`。
- 消费者取帧后，计算 `(当前时间 - 采集时间) × 1000`，得到端到端延迟（E2E Latency）。
- 实测画面实时显示：帧号、E2E 延迟、处理耗时。

**4. 上午测试结果**
- 视频源：MacBook FaceTime 摄像头，1920×1080 @ 30 FPS。
- 运行时长：37.7 秒。
- 结果：

| 指标 | 数值 |
|------|------|
| 总帧数 | 1124 |
| 总时间 | 37.7 s |
| 平均 FPS | **29.9** |

- 结论：纯采集阶段可达到接近摄像头原生帧率（30 FPS），多线程框架本身开销极小。

### 三、下午：YOLOv10 检测器集成

**1. Letterbox 预处理**
- 保持原始图像宽高比，按比例缩放至 640×640 画布内。
- 不足部分用灰色（114, 114, 114）填充，避免直接拉伸导致目标形变。
- 记录缩放比例 `scale` 与填充偏移 `ph, pw`，用于后续坐标映射。

**2. YOLOv10-N ONNX 推理**
- 使用 ONNX Runtime `CPUExecutionProvider` 加载 `yolov10n.onnx`。
- 输入：经过 Letterbox 与归一化（/255）的 `[1, 3, 640, 640]` FP32 张量。
- YOLOv10 无 NMS 设计（One-to-One Matching），输出直接可用，无需额外非极大值抑制。

**3. 坐标映射（640×640 → 原始图像）**
- 检测框坐标 `(x1, y1, x2, y2)` 先减去填充偏移 `ph, pw`。
- 再除以缩放比例 `scale`，映射回原始图像坐标系。
- 公式：
  ```
  x_orig = (x_640 - pw) / scale
  y_orig = (y_640 - ph) / scale
  ```

**4. 完整检测流水线**
- 采集 → 取帧 → Letterbox 预处理 → ONNX 推理 → 坐标映射 → 画框/标签 → 显示。
- 画面叠加信息：帧号、检测数量、当前推理延迟、平均推理延迟、端到端延迟。

**5. 下午测试结果**
- 视频源：MacBook FaceTime 摄像头，1920×1080。
- 运行时长：122.5 秒。
- 结果：

| 指标 | 数值 |
|------|------|
| 总帧数 | 1699 |
| 总时间 | 122.5 s |
| 平均 FPS | **13.9** |
| 平均推理延迟 | **52.8 ms** |

- 结果分析：
  - 平均推理延迟 52.8 ms，与 Day 1 基准测试（~50 ms）基本一致，说明多线程框架本身几乎无额外开销。
  - 平均 FPS 13.9，由推理速度决定（1000ms / 52.8ms ≈ 18.9 理论上限，实际受预处理、画框、显示等开销影响）。
  - 采集端 30 FPS，推理端 13.9 FPS，帧丢弃策略持续生效，系统运行稳定，无内存泄漏或卡顿。

### 四、今日关键结论

| 模块 | 核心技术 | 效果 |
|------|----------|------|
| 异步视频捕获 | 生产者-消费者 + 帧丢弃 | 采集与推理解耦，保持实时性 |
| Letterbox 预处理 | 等比缩放 + 灰色填充 | 保持目标形状，避免拉伸失真 |
| 坐标映射 | 逆变换还原原始坐标 | 检测框精准对应原图位置 |
| 完整流水线 | 采集→预处理→推理→后处理→显示 | 端到端延迟可控，运行稳定 |

- **核心认知**：Day 1-3 解决"模型能不能跑"，Day 4 解决"模型能不能在视频流里持续跑"。多线程架构是嵌入式实时系统的关键基础。

### 五、遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Mac 摄像头首次打开无画面 | macOS 隐私权限限制，Python 未获得摄像头访问权 | 在系统设置 → 隐私与安全性 → 摄像头中，允许终端/VS Code 访问摄像头 |
| 无摄像头时无法测试 | 部分场景不方便使用摄像头 | 在 `AsyncVideoCapture` 中加入 `use_fake` 回退模式：当摄像头打开失败时，自动生成随机彩色帧，保证多线程逻辑仍可验证 |
| 终端误输入 `qq` | 习惯性输入 | 按 `Ctrl+C` 强制中断，或按 `Q` 正常退出 |
