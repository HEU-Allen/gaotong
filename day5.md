## 五、cProfile 性能瓶颈分析

为了进一步分析系统耗时瓶颈，编写并运行了 `profile_edge_ai.py`，使用 Python 内置的 `cProfile` 对完整单帧处理流程进行性能分析。

测试设置如下：

| 项目 | 内容 |
|------|------|
| 测试帧数 | 100 |
| 输入图像 | 随机生成 480×640 图像 |
| 模型 | YOLOv10-N ONNX |
| 推理后端 | ONNX Runtime CPU |

cProfile 基础结果如下：

| 指标 | 结果 |
|------|------|
| 总耗时 | 5.10 s |
| 平均单帧耗时 | 51.02 ms |
| 平均 FPS | 19.60 |

主要函数耗时如下：

| 函数 | 累计耗时 | 说明 |
|------|------:|------|
| `session.run()` | 4.986 s | ONNX 模型推理，主要瓶颈 |
| `process_one_frame()` | 5.095 s | 单帧完整处理流程 |
| `preprocess()` | 0.103 s | 图像预处理，平均约 1.03 ms/帧 |
| `astype()` | 0.051 s | NumPy 类型转换和归一化相关操作 |
| `cv2.resize()` | 0.011 s | 图像缩放耗时较低 |

从结果可以看出，100 帧总处理时间约为 5.10 s，其中 ONNX Runtime 的 `session.run()` 累计耗时达到 4.986 s，占主要部分。相比之下，预处理部分 `preprocess()` 累计耗时只有 0.103 s，平均每帧约 1 ms。

因此，当前系统的主要瓶颈是模型推理阶段，而不是图像预处理或 OpenCV 缩放阶段。后续若要明显提升 FPS，应优先考虑更高效的推理后端，例如 TensorRT、RKNN 或 Core ML，而不是只优化 Python 预处理代码。

## 六、NumPy 向量化优化预处理

在图像预处理阶段，采用 NumPy 向量化方式完成图像归一化和维度转换，避免使用 Python 层面的逐像素循环。

核心代码如下：

```python
tensor = pad.astype(np.float32) / 255.0
tensor = np.transpose(tensor, (2, 0, 1))[np.newaxis, :]
```

这段代码完成了三个操作：

1. 将 `uint8` 图像转换为 `float32`。
2. 将像素值从 0-255 归一化到 0-1。
3. 将图像格式从 HWC 转换为 NCHW，满足 ONNX 模型输入要求。

如果使用 Python 循环逐像素处理，会带来较大的解释器开销；而 NumPy 底层使用 C 实现矩阵运算，因此处理速度更快。结合 cProfile 结果可知，当前 `preprocess()` 平均每帧约 1 ms，说明预处理阶段已经不是主要性能瓶颈。

## 七、320 / 416 / 640 分辨率权衡测试

为了分析输入画面分辨率对系统性能的影响，编写了 `resolution_benchmark.py`，分别测试 320×320、416×416 和 640×640 三种输入画面分辨率。

需要说明的是，当前 `yolov10n.onnx` 模型输入尺寸固定为 640×640。因此，本次实验测试的是“原始输入画面分辨率”变化对系统性能的影响，而不是模型输入尺寸真正变为 320 或 416。三种输入画面最终都会经过 Letterbox 预处理后送入 640×640 的 ONNX 模型。

测试结果如下：

| 输入画面分辨率 | 平均延迟 | 推理耗时 | P95 延迟 | 平均 FPS |
|------|------:|------:|------:|------:|
| 320×320 | 52.675 ms | 50.599 ms | 55.003 ms | 18.995 |
| 416×416 | 52.699 ms | 50.649 ms | 54.895 ms | 18.986 |
| 640×640 | 51.385 ms | 50.208 ms | 53.885 ms | 19.479 |

从结果来看，三种输入画面分辨率的平均延迟都在 51-53 ms 左右，平均 FPS 都接近 19 FPS，没有出现明显差异。这是因为模型推理阶段始终使用固定的 640×640 输入，主要计算量没有变化。

本次实验说明，在当前固定输入 ONNX 模型下，仅改变原始画面分辨率不能显著提升推理速度。如果希望真正比较 320、416 和 640 模型输入尺寸对速度和精度的影响，需要重新导出不同输入尺寸的 ONNX 模型，或使用支持动态输入尺寸的模型。

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| `NameError: name 'cat' is not defined` | 把终端命令误复制进 Python 文件 | 删除 `cat > ...` 和末尾 `PY`，只保留 Python 代码 |

今天完成了第一周综合项目的核心实践，把前几天学习到的模型导出、ONNX 推理、视频输入和性能分析内容整合到一个完整系统中。
